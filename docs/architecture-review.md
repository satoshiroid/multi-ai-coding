# 企画再評価 — マルチAIエージェント開発環境の穴の分析

2026-06 のライブテスト（M1 MacBook Air 8GB + Blender 実機 + Ollama/qwen2.5-coder）の
実測結果に基づき、当初企画を再点検した。結論: **コア構造（ハブ&スポーク / 3層HMAO /
HITL / 共有コンテキスト）は維持。LLM割当とエスカレーション契約を実態に合わせて改訂する。**

---

## 1. 確定した穴（実測ベース）

### 1-1. L3 = ローカルLLM前提の崩壊【最重要・実証済み】

当初のコスト戦略は「L3ワーカー（ツール操作）の代替 = ollama/qwen2.5-coder で完全無料」
だったが、8GBクラスのMacでは成立しない：

| 観測事項 | 詳細 |
|---|---|
| 7bモデルが載らない | M1 Air の利用可能VRAM ≈ 5.3GB。qwen2.5-coder:7b は常用不可 |
| 3bは bpy API を知らない | `mesh.energy`（存在しない属性）、2次元 `Vector()`、`import math` 漏れを毎回生成 |
| 自己デバッグが機能しない | エラーをフィードバックしても同じ誤りを再生成（2回連続で同一エラー） |
| プロンプト/サニタイザーで救えない | API ルール明記・正規表現パッチを重ねても新種のエラーが出続ける |
| 熱問題 | ファンレスM1で約20 tok/s の推論を数分続けると深刻な発熱 |

**対応**: ローカルLLMをクリティカルパスから除外し、オプション実験（`--builder ollama`）に
格下げ。ツール操作は Claude（クラウド）へ。

### 1-2. confidence_score 自己申告の弱さ

エスカレーション契約（`confidence_score < 70` で L2 介入）は LLM の**自己申告**が根拠。
実測では小型モデルは誤ったコードに高 confidence を返すため、シグナルとして機能しない。

**対応（フェーズB）**: 客観シグナルを一次トリガーへ再定義する。

| ドメイン | 客観シグナル |
|---|---|
| design (Blender) | スクリプト実行エラー / レンダPNG有無 / vision レビュースコア |
| mecha (FreeCAD) | スクリプト実行エラー / STEP エクスポート成否 / 寸法計測スクリプト |
| circuit (KiCAD) | **ERC / DRC**（ツール内蔵検証 — LLM評価より確実） |
| software | lint / ユニットテスト実行結果 |
| ドメイン間 | 既存の数値整合性チェック（内寸 vs 基板外寸） |

confidence_score は補助シグナルとして残す（契約スキーマは互換維持）。

### 1-3. 検証ループが Blender にしか無い

vision レビューループは design ドメインのみ。FreeCAD/KiCAD は数値整合チェックだけで、
成果物そのものの妥当性検証が無い。上表のツールベース検証ゲートを各ドメインに追加する
（段階導入: Blender → FreeCAD → KiCAD）。

### 1-4. APIキーの全漏洩【要即時対応】

開発セッション中に ANTHROPIC_API_KEY / GEMINI_API_KEY / DISCORD_BOT_TOKEN が
チャットログ・ターミナルスクリーンショットに露出した。**全キーの再発行が稼働の前提条件**。

- Anthropic: https://console.anthropic.com → API Keys → 旧キー無効化+新規発行
- Gemini: https://aistudio.google.com/apikey
- Discord: Developer Portal → Bot → Reset Token
- 再発防止: `.env` は `.gitignore` 済みを維持。キーを `echo`/`cat` でターミナルに表示しない

### 1-5. Gemini 無料枠のレート制限リスク

全層を gemini-2.0-flash に集中させると、9ステージ+修正ループで 429 が頻発しうる
（指数バックオフは実装済みだが根本対策ではない）。L3 を Claude に分離することで
Gemini 呼び出し数自体が減る。

### 1-6. MCPサーバー成熟度の楽観視

blender-mcp ですらポート9876の実態は **MCP/SSE ではなく raw JSON-over-TCP** であり、
専用クライアント（`src/mcp/blender_tcp.py`）の自作が必要だった。freecad-mcp /
kicad-mcp も同等の適応作業（プロトコル検証・アダプタ実装）を見込むこと。

### 1-7. 同時リソース制約

Blender + FreeCAD + KiCAD + Bot/API サーバーの同時起動は 8GB では非現実的。
パイプラインはステージ単位でツールを順次起動・終了する運用とする。
Ollama 常駐の廃止で約3〜4GBのメモリが解放される。

### 1-8. コスト試算（Claude L3 移行後）

Claude Opus 4.8: $5 / $25 per 1M トークン（入力/出力）。

| 単位 | 概算 |
|---|---|
| bpy スクリプト1生成（~2K in / ~2K out） | ≈ $0.06 |
| Blenderモデリング1セッション（3往復+修正） | ≈ $0.3〜0.5 |
| フルパイプライン1回（全ドメイン） | ≈ $1〜3 |

3bローカルは「無料」だが、エラー再試行の繰り返しで成果物に到達できなかった。
**到達率を含めた実質コストは Claude の方が低い。**
節約したい場合は `--builder-model claude-haiku-4-5`（約1/5のコスト）も選択可。

### 1-9. アプリ開発パイプラインの検証欠如（将来フェーズ）

APP_PIPELINE の MVP 実装ステージは生成コードを実行検証しない。1-2 と同じ
「ツールで検証」原則（pytest / lint をMCP経由またはサブプロセスで実行）を適用する。

## 2. 妥当性を再確認した部分（変更なし）

- **ハブ&スポーク**: ワーカー間直接通信禁止 — 制約違反増幅防止の根拠は変わらない
- **3層HMAO**: L2の介入価値は「別コンテキストでのレビュー」として再定義しても有効
- **HITLゲート / 72hタイムアウト / SQLite状態永続化**: モックE2Eで動作確認済み
- **共有コンテキスト（寸法overwrite / BOMマージ）と整合性チェック**: 数値検証は健在
- **hardware/app 自動判別の2パイプライン構成**: 当初計画からの拡張だが概念は不変

## 3. 改訂モデル割当

| 層 | 役割 | モデル | 理由 |
|---|---|---|---|
| L1 / L2 | 計画・分解・調停（JSON系） | gemini-2.0-flash（無料枠） | 構造化タスクは無料枠で十分 |
| **L3** | **ツール操作スクリプト生成** | **claude-opus-4-8**（fallback: gemini-2.0-flash） | コード精度が決定的。再試行減で実質コスパ最良 |
| レビューア | vision採点・修正指示 | gemini-2.0-flash（無料枠・vision対応） | 採点+自然言語指示で十分 |
| ローカルLLM | — | クリティカルパスから除外 | 1-1 参照。16GB+マシンでは再評価可 |

## 4. 段階導入ロードマップ

1. **フェーズA（実施済み）**: Blender ライブを Claude ビルダー化、settings.yaml L3 変更
2. **フェーズB**: エスカレーション条件の客観化（実行エラー/検証ゲートを一次トリガーに）
3. **フェーズC**: FreeCAD ライブ統合（freecad-mcp 検証 + STEP/寸法検証ゲート）
4. **フェーズD**: KiCAD ライブ統合（ERC/DRC を承認ゲート前の必須チェックに）
5. **フェーズE**: アプリパイプラインの実行検証（pytest/lint ゲート）
